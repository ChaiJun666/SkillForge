# SkillForge AGENTS

Author: wwj
Version: v1.0
Date: 2026-05-09

## Project Overview

SkillForge is a desktop IDE for AI Agent Skill development. It focuses on creating, editing, validating, debugging, packaging, and eventually distributing Agent Skills across ecosystems such as Claude Skills, OpenAI Agent Skills, Codex Skills, Cursor Rules, MCP Tool Skills, and custom agent frameworks.

The product direction is an IDE-like workspace for Agent Skills, with dense developer tooling rather than a marketing-style web app.

## Product Priorities

MVP scope is:

- Skill Explorer
- Skill Editor
- YAML metadata form
- Markdown editor
- Skill Validator
- Script Runtime
- Import and export

Later phases include Skill Simulator, AI Generator, Skill Search, Marketplace, Workflow Graph, and Cloud Sync.

## Architecture Direction

Treat the app as a local-first desktop IDE:

- Keep filesystem operations, process execution, and SQLite access behind Tauri/Rust boundaries.
- Keep editor, explorer, preview, validation UI, and terminal UI in React.
- Prefer typed contracts between React and Tauri commands.
- Keep Skill parsing and validation deterministic before adding AI-assisted flows.
- Do not couple marketplace or cloud concepts into MVP-local editing paths.

## Frontend Conventions

- Build the first screen as the actual IDE workspace: sidebar, explorer, editor, preview, and terminal surfaces.
- Use dense, work-focused UI. Avoid landing pages, decorative hero layouts, and oversized marketing sections.
- Use shadcn/ui primitives where they fit, with TailwindCSS for layout and local styling.
- Use lucide icons for toolbar and command buttons when available.
- Keep panels stable in size; editor, explorer, preview, and terminal layout should not jump when content changes.
- Monaco should own code/Markdown editing behavior; avoid hand-rolled editor behavior unless the existing editor APIs cannot cover it.

## Backend And Runtime Conventions

- Use Rust/Tauri commands for local filesystem access, script execution, SQLite storage, and OS integration.
- Validate user-selected paths and never execute scripts outside the intended workspace without explicit user action.
- Script Runtime must capture stdout, stderr, exit status, duration, timeout, and environment in a structured way.
- Avoid silent fallback behavior for validation and runtime errors. Surface actionable errors to the UI.

## Skill Model

Use the PRD data model as the initial contract:

```ts
interface Skill {
  id: string
  name: string
  description: string
  version: string
  compatibility: string[]
  tags: string[]
  path: string
  createdAt: string
  updatedAt: string
}

interface SkillExecution {
  id: string
  skillId: string
  input: string
  output: string
  logs: string[]
  duration: number
  status: 'success' | 'failed'
}
```

Keep these interfaces synchronized with Rust command payloads and SQLite schema as implementation starts.

## Validation Scope

Skill validation should cover:

- Required `SKILL.md` file
- Supported optional folders such as `scripts/`, `references/`, `assets/`, and `tests/`
- YAML frontmatter syntax and required metadata fields
- Markdown prompt structure
- Compatibility and tag constraints
- Script path and runtime safety checks

Prefer schema-driven validation for frontmatter where practical.
